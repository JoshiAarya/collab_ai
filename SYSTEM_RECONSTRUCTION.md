## 1. SYSTEM PURPOSE (STRICT)

### What it solves
CollabAI is a real-time collaborative workspace where teams discuss projects in threaded "discussions" and invoke an AI assistant (`@CollabAI`) inline in chat. The AI has access to project context (documents, prior decisions, blockers, topics, action items) and responds with that context injected into its prompt. Over time, the system extracts structured engineering knowledge (decisions, blockers, topics, action items) from conversations and surfaces them on a dashboard.

**Stack:** React 19 + Vite 7 · Node.js 20 / Express 5 · MongoDB 7 · Groq API · OpenAI / Anthropic / Google (user-supplied keys) · Xenova local embeddings (384-dim, no API cost)

### What it explicitly does NOT solve
- It does NOT provide real-time collaborative document editing (no CRDT, no OT).
- It does NOT support streaming AI responses. All AI responses are delivered as a single complete message after generation.
- It does NOT support DeepSeek at runtime. The `callProvider` switch has no `deepseek` case — falls through to `throw new Error('Unsupported provider: deepseek')`. Groq, OpenAI, Anthropic, and Google are all fully implemented with real SDK calls.
- It does NOT persist WebSocket authentication state across reconnects. Each reconnect requires re-sending `auth` + `join-project`.
- It does NOT support file uploads via multipart form. Documents are uploaded as raw text strings in JSON body (max 10MB body limit). Only `.txt` and `.md` files are accepted in the UI.
- It does NOT support PDF parsing. `fileType: 'pdf'` is stored but content is treated as plain text.
- It does NOT support pagination for messages in WebSocket flow. `getDiscussionMessages` fetches last 50 on join.
- It does NOT support message deletion or editing.
- It does NOT have a notification system.
- It does NOT support xAI (Grok). Listed in ModelSelector UI as `comingSoon: true` — clicking does nothing.

## 2. GLOBAL INVARIANTS

1. Every `Project` has exactly one `Discussion` with `isMain: true`, created atomically in `projectService.createProject`. If this document is missing, `join-project` WebSocket events will load an empty message list and the main thread will not appear in the sidebar.

2. `ProjectState` has a unique index on `projectId`. There is exactly zero or one `ProjectState` per project. The system uses `ProjectState.exists({ projectId })` as the gate to decide whether to use the entity-aware context path or the legacy path.

3. `Topic.normalizedName` is unique per `projectId` (compound unique index). Two topics with the same normalized name cannot coexist in the same project.

4. `DocumentChunk.embedding` must be exactly 384 dimensions (enforced by Mongoose validator). The embedding model is `Xenova/all-MiniLM-L6-v2` (local, no API key).

5. JWT tokens expire in 7 days. `authService.js` uses the secret `process.env.JWT_SECRET || 'collab-ai-secret-key-change-in-production'`. `config/index.js` exposes `process.env.JWT_SECRET || 'dev-secret-change-in-production'` — these two fallbacks differ. If `JWT_SECRET` is not set in env, tokens generated by `authService` may not verify via `config.jwtSecret`.

6. Rate limiting is in-memory only (not Redis). On server restart, all rate limit counters reset. Limits: 20 AI requests/minute per user, 50 AI requests/minute per project (configurable via `RATE_LIMIT_USER_PER_MIN` / `RATE_LIMIT_PROJECT_PER_MIN`). WebSocket message rate: 30 messages/minute per connection (configurable via `WS_RATE_LIMIT_MAX_MESSAGES`).

7. `EncryptionService` is a singleton that throws in its constructor if `ENCRYPTION_KEY` is absent or not a valid 64-char hex string. This means the entire Node.js process crashes at startup if the env var is wrong. There is no graceful degradation.

8. The extraction pipeline (`InsightExtractor` + `KnowledgeAggregator`) is always non-blocking and non-critical. Any failure is caught, logged as a warning, and does not affect message delivery.

9. Messages are ordered by `timestamp` (a Number, `Date.now()`) ascending in `getDiscussionMessages`. The query sorts descending then reverses the array in memory.

10. `Project.activeLLM.provider = 'server'` is mapped to `'groq'` in `AIOrchestrator.selectModel`. The model `'server'` is mapped to `'llama-3.1-8b-instant'`.

11. The extraction pipeline ONLY runs on the main discussion (`isMain: true`). Parallel discussions feed the knowledge model exclusively via summaries. `_triggerExtractionForMessage` fetches the discussion and returns early if `isMain === false`.

12. Every Groq call creates a fresh `new Groq({ apiKey })` client. There is no persistent connection, no session, no conversation state on Groq's side between calls. Switching projects cannot bleed context.

13. `config/index.js` validates that `GROQ_API_KEY` or `CHATBOT_API_KEY` is present at startup. If missing, the server throws and refuses to start.

14. `EncryptionService.decryptForUse` is called on the Groq API key every time `getServerGroqKey()` is called. If the key is plaintext (not encrypted), it logs a warning and returns it as-is (backward compatibility).

## 3. RUNTIME ENTRY POINTS

### 3.1 HTTP Endpoints

All routes are mounted under `/api`. All project routes require `authenticate` middleware (JWT Bearer token). Auth routes are public.

#### POST /api/auth/register
- Body: `{ username, email, password }`. Validation: username 2–50 chars, email regex, password 6–100 chars.
- Execution: checks duplicate → bcrypt hash (10 rounds) → saves User → generates JWT → sends welcome email (non-blocking).
- Output: `{ success: true, user: sanitizedUser, token }`.

#### POST /api/auth/login
- Body: `{ email, password }`. Identical error message for "not found" and "wrong password" (intentional).
- Sets `user.isOnline = true`, `user.lastSeen = new Date()` on success.

#### GET /api/auth/verify
- Decodes JWT → returns user without password. 401 if token invalid or user deleted.

#### GET /api/auth/google, GET /api/auth/google/callback
- Full OAuth flow. Redirects to `FRONTEND_URL?token=JWT&provider=google` on success.

#### POST /api/auth/google (direct)
- If `idToken` present: verifies via Google tokeninfo endpoint. If no `idToken`: uses `googleId` + `email` directly — NO verification (security risk).

#### POST /api/projects
- Body: `{ title, problemStatement }`. Creates project + main discussion + updates User.projects.
- `activeLLM` defaults to `{ provider: 'server', model: 'llama-3.1-8b-instant' }`.
- `inviteCode` = 8-char hex via `crypto.randomBytes(4).toString('hex')`.

#### GET /api/projects
- Returns `Project.find({ 'members.userId': userId })` sorted by `updatedAt` desc.

#### GET /api/projects/:projectId
- Checks membership (403 if not member) → returns populated project.

#### POST /api/projects/join
- Body: `{ inviteCode, discussionId? }`. Joins project + optionally joins specific discussion.
- Output: `{ success, project, discussionId, alreadyMember, addedToDiscussion }`.
- On join: adds user to project members AND to the main discussion participants.

#### POST /api/projects/invite-preview
- Returns project info + membership status + optional discussion info without joining.

#### POST /api/projects/:projectId/invite-email
- Sends project invite email via SendGrid. Requires project membership (not owner-only).

#### PATCH /api/projects/:projectId/stage
- Owner only. Updates `Project.stage` (manual, independent of `ProjectState.stage`).

#### PATCH /api/projects/:projectId
- Owner only. Updates `activeLLM` and/or `stage`.

#### PUT /api/projects/:projectId/llm
- Owner only. Updates `activeLLM`.

#### POST /api/projects/:projectId/api-key
- Owner only. Stores provider API key in `Project.apiKeys` Map. NOTE: stored plaintext despite UI saying "stored encrypted per project".

#### GET /api/projects/:projectId/discussions
- Member only. Returns all active discussions sorted `isMain DESC, lastActivity DESC`.
- Enriches each non-main discussion with `latestSummary: { _id, createdAt, messageCountAtSummary }` for stale summary detection.

#### POST /api/projects/:projectId/discussions
- Member only. Creates parallel discussion (`isMain: false`). Creator + owner auto-added as participants.

#### POST /api/projects/:projectId/discussions/:discussionId/invite
- Owner or participant only. Adds a project member to a discussion.

#### POST /api/projects/:projectId/discussions/:discussionId/invite-email
- Owner or participant only. Sends email with invite code + discussionId.

#### GET /api/projects/:projectId/documents
- Member only. Returns all project documents with chunk count (not actual chunk data).

#### GET /api/projects/:projectId/documents/:documentId/chunks
- Member only. Debug endpoint. Returns chunk metadata (content, embedding dimensions, first 5 embedding values).

#### POST /api/projects/:projectId/documents
- Member only. Body: `{ title, content, fileType }`. Saves Document immediately, fires async embedding generation.
- `fileType` normalized: anything containing 'pdf' → 'pdf', otherwise → 'text'.

#### GET /api/projects/:projectId/summaries
- Member only. Returns all project summaries.

#### POST /api/projects/:projectId/discussions/:discussionId/summarize
- Member only. Cannot summarize main discussion (400).
- Generates summary via LLM → saves with `messageCountAtSummary = discussion.messageCount`.
- Triggers intelligence pipeline on summary content (non-blocking). Source is `isAI: true` → decisions get `needsHumanValidation: true`.

#### PUT /api/projects/:projectId/discussions/:discussionId/summaries/:summaryId
- Member only. Regenerates summary with custom prompt.

#### DELETE /api/projects/:projectId/discussions/:discussionId/summaries/:summaryId
- Member only. Deletes summary.

#### GET /api/projects/:projectId/dashboard
- Owner only (403 for non-owners). Calls `buildDashboardResponse(projectId)`.

#### POST /api/projects/:projectId/dashboard/refresh
- Owner only. Force-runs extraction on main discussion (bypasses rate gate and overlap guard), then calls `buildDashboardResponse(projectId)`.
- Sparsity guard: skips if main discussion has < 3 messages or < 150 total chars.

#### GET /api/user/profile, PUT /api/user/profile, PUT /api/user/password, GET /api/user/stats
- Standard CRUD on authenticated user's own profile.

#### GET /health
- Returns server health: DB status, WS connection count, uptime, version.

#### GET /docs
- Swagger UI (OpenAPI 3.0).

#### GET /api/system/info
- Development only. Returns config, WS stats, memory usage.

### 3.2 WebSocket Events

Single `WebSocketServer` attached to the HTTP server. No path prefix.

#### Connection lifecycle
1. Client opens WebSocket → server assigns `clientId`, stores metadata.
2. Client sends `{ type: 'auth', token: JWT }`.
3. Client sends `{ type: 'join-project', projectId, discussionId }`.
4. Server verifies membership (skipped if userId null), loads last 50 messages, sends `discussion-joined`.
5. Heartbeat: server pings every `WS_HEARTBEAT_INTERVAL` ms (default 30s). Dead connections terminated.

#### Message types (client → server)
| Type | Payload |
|------|---------|
| `auth` | `{ token }` |
| `join-project` | `{ projectId, discussionId }` |
| `project-chat` | `{ text }` |
| `ping` | — |

#### Message types (server → client)
| Type | Payload |
|------|---------|
| `auth-success` | `{ user: { userId, username } }` |
| `discussion-joined` | `{ messages: [...], discussionId }` |
| `project-chat` | `{ message: { user, text, time, isAI } }` |
| `ai-thinking` | `{ status: 'generating' }` |
| `ai-error` | `{ message }` |
| `pong` | `{ timestamp }` |
| `error` | `{ message }` |
| `server-shutdown` | `{ message }` |

### 3.3 Background Jobs
None. All processing is event-driven. Scripts in `src/scripts/` are one-off maintenance tools run manually.

## 4. COMPLETE DATA FLOW SIMULATIONS

### Flow 1: User sends message with @CollabAI

1. User types `@CollabAI question` → frontend sets `aiThinking = true` → sends `{ type: 'project-chat', text }` over WebSocket.
2. Server `handleProjectChat`: rate limit check → `discussionService.addMessage` → broadcast to all clients in discussion → `_triggerExtractionForMessage` (non-blocking, returns early because text starts with `@CollabAI`) → `handleAIInvocation`.
3. `handleAIInvocation`: broadcasts `ai-thinking` → `aiService.generateResponse` → `AIOrchestrator.handleRequest`.

**Inside AIOrchestrator.handleRequest:**

4. `requestId = randomUUID()`.
5. `RateLimiter.checkLimits(userId, projectId)` — 20/min user, 50/min project. Throws `RateLimitError` if exceeded.
6. `selectModel`: maps `'server'` → `'groq'`, `'server'` model → `'llama-3.1-8b-instant'`.
7. `ProjectState.exists({ projectId })` → entity-aware path or legacy path.

**Entity-aware context (`buildEntityAwareContext`):**

8. 10 parallel MongoDB queries: ProjectState, last 5 active Decisions, all open Blockers (severity desc), top 8 Topics (status='active' — BUG: returns nothing, should be 'stable'), top 5 non-completed ActionItems, last 3 Summaries for current discussion, last 30 Messages, Project, Discussion, all Discussions.
9. Document retrieval: semantic (VectorStore cosine similarity, top 5 chunks) or positional fallback (first 3 docs, 2000 chars each).
10. Parallel discussion summaries: for each non-current discussion, fetches latest summary. Populates `context.discussions` array.

**System prompt construction (`_buildEntitySystemPrompt`):**

11. Layer order: base instruction → project metadata → discussion metadata → Layer 1 (pinnedContext or stage+momentum) → Layer 2 (decisions) → Layer 3 (blockers) → Layer 4 (topics — always empty due to bug) → Layer 5 (documents) → Layer 6 (actions) → Layer 7 (parallel discussion summaries).

**Message array (`constructMessages`):**

12. `[{ role: 'system', content: systemPrompt }]` + recent messages (skipping `user === 'System'`, mapping CollabAI → assistant) + `{ role: 'user', content: prompt }`.

**Provider call:**

13. `TokenManager.countMessagesTokens` → `LLMGuardrails.guardedCall` → validates token count ≤ model limit → `executeWithRetry` → `executeWithTimeout(90s)` → `callGroq` / `callOpenAI` / `callAnthropic` / `callGoogle`.
14. `callGroq`: `new Groq({ apiKey })` (fresh client every call, stateless) → `client.chat.completions.create({ model, messages, temperature: 0.7, max_tokens: 8192 })`.
15. Returns response string.

**Post-response extraction (non-blocking):**

16. `InsightExtractor.extractFromAIResponse` → window of last 13 human messages (AI text excluded) → overlap guard → LLM extraction → 4-layer validation → `KnowledgeAggregator.mergeInsights`.
17. AI-sourced decisions get `needsHumanValidation: true`.

18. AI message saved → broadcast → frontend sets `aiThinking = false`.

### Flow 2: User sends regular message (extraction pipeline)

1. User sends message (not `@CollabAI`, ≥ 30 chars).
2. Server saves message → broadcasts → `_triggerExtractionForMessage` (non-blocking).
3. `_triggerExtractionForMessage`:
   - Fetches discussion → if `isMain === false`: returns immediately (parallel discussions never trigger extraction).
   - `InsightExtractor.extractFromMessage`: increments in-memory counter per discussionId. If counter < 5 (`MIN_MESSAGES_BETWEEN_EXTRACTIONS`): skips. Resets to 0 when threshold reached.
   - `_buildWindow(discussionId, 14)` → last 14 messages (excluding System).
   - Overlap guard: if ≥ 90% of window message IDs already in `supportingMessageIds` of existing artifacts → skip.
   - LLM extraction call (temperature 0.1, maxTokens 1000).
   - 4-layer validation.
   - `KnowledgeAggregator.mergeInsights` → `_recomputeProjectState`.

### Flow 3: Parallel discussion summarization

1. User clicks "Summarize" in three-dot menu of a non-main discussion.
2. `POST /api/projects/:projectId/discussions/:discussionId/summarize`.
3. Fetches last 50 messages → LLM generates summary → saves `Summary` with `messageCountAtSummary = discussion.messageCount`.
4. Intelligence pipeline runs on summary content (non-blocking): `InsightExtractor.extractFromMessage` with `isAI: true` → `KnowledgeAggregator.mergeInsights`. Decisions from this extraction get `needsHumanValidation: true`.
5. Frontend: `GET /api/projects/:projectId/discussions` returns `latestSummary.messageCountAtSummary`. If `messageCount - messageCountAtSummary >= 10`: shows ⚠️ stale indicator in sidebar.
6. AI context: `buildEntityAwareContext` fetches latest summary per parallel discussion → populates Layer 7 (Parallel Discussion Summaries) in system prompt.

### Flow 4: Dashboard generation

1. `GET /api/projects/:projectId/dashboard` (owner only) → `buildDashboardResponse(projectId)`.
2. Shared metrics: loads ALL messages from ALL discussions (no limit — O(total messages)). Computes 7-day activity histogram, participant stats, discussion breakdown, message type counts.
3. `StrategicSignalEngine.generateSignals({ projectId })` — reads from `ProjectInsights` (legacy). Always empty for entity-model projects.
4. `ProjectState.exists({ projectId })` → routes to entity, legacy, or LLM path.

**Entity model path (`buildEntityDashboard`):**
- Stable topics (limit 8), active decisions excluding `needsHumanValidation: true` sorted by `occurrenceCount desc, createdAt desc` (limit 8), confirmed/high-severity unresolved blockers (limit 8), non-completed actions (limit 8), ProjectState.
- Enriches decisions with `topicName`, `rationale`, `timestamp`.
- Enriches blockers with `daysOpen` (from `raisedAt`), `topicName`.
- Enriches actions with `assignee`, `topicName`.
- Returns `{ source: 'entity-model', topics, decisions, openQuestions, actionItems, stage, enrichedDecisions, enrichedBlockers, enrichedActions, projectState, signals, ...sharedMetrics }`.

**Dashboard refresh (`POST /api/projects/:projectId/dashboard/refresh`):**
- Owner only. Calls `InsightExtractor.forceExtractForProject` (bypasses rate gate and overlap guard).
- Sparsity guard: skips if < 3 messages or < 150 total chars.
- Merges insights → calls `buildDashboardResponse` → returns fresh dashboard.

### Flow 5: Document upload → embeddings → retrieval

1. `POST /api/projects/:projectId/documents` → saves Document → returns immediately.
2. Async: `chunkText(content, 900, 100)` → `EmbeddingService.embedBatch` (sequential, local model) → `DocumentChunk.insertMany`.
3. On next AI request: `VectorStore.count(projectId) > 0` → `EmbeddingService.embedText(prompt)` → `VectorStore.search` (loads ALL chunks into memory, cosine similarity in JS loop, returns top 5).

### Flow 6: Invite + join flow

1. Owner copies invite code (8-char hex) from Settings.
2. Invitee navigates to `?invite=CODE` or `?invite=CODE&discussion=ID`. URL parsed by `router.js` (`getInviteCodeFromUrl`, `getDiscussionInviteFromUrl`).
3. `InviteConfirmModal` → `POST /api/projects/invite-preview` → shows project/discussion info.
4. User accepts → `POST /api/projects/join` → joins project + optionally joins discussion.
5. `sessionStorage.setItem('pendingDiscussionId', discussionId)` → `ProjectWorkspace` reads it after discussions load → switches to that discussion.

## 5. AI SYSTEM BEHAVIOR (CRITICAL)

### Context construction — what gets included

**Gate:** `ProjectState.exists({ projectId })` — MongoDB existence check. Once created, entity-aware path is used forever.

**Entity-aware path layers (in order):**

| Layer | Content | Source | Limit | Notes |
|-------|---------|--------|-------|-------|
| System base | Role instruction | Hardcoded | Always | — |
| Project metadata | title, description, memberCount, discussionCount, stage | Project | Always | — |
| Discussion metadata | title, isMain | Discussion | Always | — |
| Layer 1 | pinnedContext OR stage+momentum+counts | ProjectState | Always | pinnedContext preferred |
| Layer 2 | Active decisions | Decision | Last 5, timestamp desc | — |
| Layer 3 | Open blockers | Blocker | All unresolved, severity desc | — |
| Layer 4 | Active topics | Topic (status='active') | Top 8 | BUG: 'active' not valid, always empty |
| Layer 5 | Relevant documents | DocumentChunk or Document | Top 5 chunks or 3 docs | Semantic or positional |
| Layer 6 | Pending actions | ActionItem | Top 5 non-completed | — |
| Layer 7 | Parallel discussion summaries | Summary | Latest per discussion | Only if summary exists |
| Layer 8 | Current discussion summaries | Summary | Last 3 | Via constructMessages |
| Layer 9 | Recent messages | Message | Last 30 | Via constructMessages |

### Groq connection model

Every call to `callGroq` creates `new Groq({ apiKey })` — a fresh client. Groq's API is stateless REST. There is no session, no persistent connection, no conversation history on Groq's side. Switching projects cannot bleed context because each request builds its context from scratch, scoped to the exact `projectId` + `discussionId`.

### Token overflow behavior

`TokenManager.trimContext` is defined with real logic but is NEVER called from `callProvider`. If token count > model limit (e.g. 7200 for llama-3.1-8b-instant): `LLMGuardrails.validateRequest` returns `{ valid: false }` → error thrown → shown in chat as System message. No automatic trimming occurs.

### Model selection

- `provider === 'server'` → `'groq'`
- `model === 'server'` or falsy → `'llama-3.1-8b-instant'`
- Groq, OpenAI, Anthropic, Google: fully implemented with real SDK calls.
- DeepSeek: not in the switch — falls through to `throw new Error('Unsupported provider: deepseek')`.
- xAI: not in the switch — same error.

**Supported models per provider (LLMGuardrails):**
- groq/server: `llama-3.1-8b-instant`, `llama-3.1-70b-versatile`, `llama-3.3-70b-versatile`, `llama-3.1-405b-reasoning`, `mixtral-8x7b-32768`, `gemma-7b-it`, `gemma2-9b-it`
- openai: `gpt-4.1`, `gpt-4.1-mini`, `gpt-4.1-nano`, `gpt-4o`, `gpt-4o-mini`, `o3`, `o4-mini`
- anthropic: `claude-opus-4-6`, `claude-sonnet-4-6`, `claude-haiku-4-5-20251001`
- google: `gemini-2.5-pro`, `gemini-2.5-flash`, `gemini-2.0-flash`, `gemini-2.0-flash-lite`

**Provider call implementations:**
- `callGroq`: `new Groq({ apiKey })` → `chat.completions.create`. Standard OpenAI-compatible format.
- `callOpenAI`: `new OpenAI({ apiKey })` → `chat.completions.create`. Same format.
- `callAnthropic`: `new Anthropic({ apiKey })` → `messages.create`. System prompt extracted as top-level param. Chat messages filtered to non-system.
- `callGoogle`: `new GoogleGenerativeAI(apiKey)` → `startChat({ history })` → `sendMessage`. Converts OpenAI-style messages to Gemini format. System prompt prepended to first user message if history is empty.

### AbortController timeout

`executeWithTimeout(callFn, 90000)` (90 seconds — raised to accommodate Gemini 2.5 Pro / Claude Opus which can take 45–60s). Creates an `AbortController` and passes `controller.signal` to `callFn`. However, none of the provider call methods (`callGroq`, `callOpenAI`, `callAnthropic`, `callGoogle`) accept or use the signal parameter. The timeout fires `controller.abort()` but the underlying HTTP request continues running. The timeout error is thrown to the caller but the request is not actually cancelled.

## 6. INTELLIGENCE PIPELINE (COMPLETE)

### Trigger conditions

| Trigger | Condition | Rate gate |
|---------|-----------|-----------|
| User message | `isMain === true`, text ≥ 30 chars, not `@CollabAI` | Every 5th message per discussion (in-memory counter, `MIN_MESSAGES_BETWEEN_EXTRACTIONS = 5`) |
| AI response | After every `handleRequest` | Overlap guard only (no rate gate) |
| Summary creation | After `POST .../summarize` | None (always runs) |
| Dashboard refresh | `POST .../dashboard/refresh` | None (force extraction, bypasses all gates) |

### Extraction prompt rules (current)

- ABSOLUTE RULE: every artifact must be directly stated in the conversation. No inference, no training knowledge.
- Decisions MUST start with a choice verb: `use / adopt / store / keep / switch to / standardize on / deploy / migrate to`.
- Decisions with indefinite article (`use a pool`) → reclassified as action.
- Action items are only extracted if they are strictly actionable and still open at the end of the conversation.
- Entity caps removed: prompt explicitly asks to extract "ALL qualifying decisions" (up to 20).
- No specific technology examples in prompt (removed to prevent hallucination).

### Supersession Detection (New)
A second LLM pass validates supersessions. If a new decision explicitly contradicts, overrides, or reverses an active decision (e.g., "instead of X we'll use Y"), this pass records the supersession, maintaining the integrity of the knowledge graph against false-positive reversals.

### 4-layer validation (InsightExtractor)

| Filter | Decisions | Blockers | Actions | Topics |
|--------|-----------|----------|---------|--------|
| `meetsMinLength` (≥15 chars) | ✓ | ✓ | ✓ | — |
| `notNoise` (not in NOISE_EXACT, ≥3 words) | ✓ | ✓ | ✓ | — |
| `appearsInConversation` (keyword match) | ✓ | ✓ | ✓ | ✓ |
| `looksLikeDecision` (commitment language) | ✓ | — | — | — |

`appearsInConversation`: extracts non-stop-word words > 3 chars. If ≤2 keywords: requires all to appear. If >2 keywords: requires ≥60% to appear.

`looksLikeDecision` patterns (9 total):
- `we('ll|will|are going to|have decided|decided|chose|are using|will use)`
- `let's (use|go with|adopt|switch to|deploy|standardize)`
- `going with`, `decided (to use|on)`
- `we('ve|have) (chosen|selected|picked|agreed|adopted)`
- `our (approach|stack|architecture|solution|choice) (is|will be)`
- `(use|adopt|store|keep|switch to|standardize on|mirror) <non-article word>`
- `we should (use|adopt|store|keep|switch to|deploy|standardize on|migrate to)`

### Overlap guard

Checks if ≥ 90% of window message IDs are already in `supportingMessageIds` across existing artifacts. If so, skips extraction. Threshold is 90% (raised from 80% to avoid skipping on long conversations). This overlap guard handles user messages as well as AI extraction responses consistently.

### KnowledgeAggregator merge pipeline

Order: Topics → Blockers → Decisions → Actions → ProjectState recompute.

**Topics:**
1. `normalizeTopicName` → strips stop words, singularizes.
2. Generate embedding.
3. Semantic dedup against existing (threshold 0.82) → if match: increment count, update lastSeenAt.
4. If no match: `findOneAndUpdate` upsert by `{ projectId, normalizedName }`.
5. Promote to `stable` when `count >= 3`.
6. Decay: `stable` → `parked` when `lastSeenAt < 30 days ago`.

**Blockers:**
1. Placeholder + noise filter.
2. Cluster check (threshold 0.85) → if match: increment occurrenceCount, add message IDs.
3. `_inferBlockerSeverity`: keyword-based override (does NOT blindly trust LLM high severity).
   - High: `blocking/blocked/crash/crashing/failing/failed/unusable/impossible/cannot deploy/data corruption/data loss/out of disk/out of memory/service down/unavailable`
   - Low: `minor/cleanup/refactor/nice to have/would be nice/eventually/optional/low priority/inconvenience`
   - Medium: `slow/complex/hard to/difficult/large memory/inconsistent/unreliable/flaky/captcha/won't scale/memory usage/forbidden/403/performance/storage/constraint/limit`
   - Default: medium (ignores LLM high unless pattern confirms)
4. Upsert by `{ projectId, text: cleanText, resolved: false }`. Blocker visibility is correctly maintained for summary-sourced blockers by bypassing `needsHumanValidation` filtering limits which formerly blocked them.

**Decisions:**
1. Noise filter → `_isLowLevelImplementationDecision` (rejects hash/parse/loop unless ARCH_SIGNALS present) → `_isImplementationAction` (reclassifies add/build/implement or `use a/an X` as action).
2. `normalizeDecisionText` → strips leading commitment phrases.
3. Substring containment dedup → merge if match.
4. Semantic similarity ≥ 0.80 → merge if match.
5. New: topic linking (hint → topicMap → fallback similarity 0.75). Blocker resolution via `resolvedBlockerHint`.
6. `_normalizeArtifactText`: strips conversational prefixes + inline fillers + capitalizes.
7. AI-sourced: `needsHumanValidation: true`. Human message that merges with it: clears flag.
8. Supersession logic applies during merge: matching existing artifacts are superseded if identified by InsightExtractor.

**Actions:** Same dedup pipeline (substring → semantic → new). `normalizeActionVerb` for normalization. Prioritizes specific engineering tasks over generic architectural monitoring statements.

**Text normalization (`_normalizeArtifactText`):**
- Strips: `we'll / let's / I think we / going with / decided to / we've chosen / our approach is / ...`
- Strips fillers: `just / basically / kind of / probably / maybe / actually / really / ...`
- Strips orphaned `to ` after prefix stripping.
- Capitalizes result. Falls back to original if empty.

### ProjectState recompute

After every `mergeInsights`. Full recompute (not incremental).

1. Topic decay (30-day rule).
2. Dashboard counts with confidence-based gating:
   - Blockers shown: `occurrenceCount >= 2` OR `severity === 'high'`
   - Decisions shown: `status: 'active'`, `needsHumanValidation: { $ne: true }`
   - Topics shown: `status: 'stable'`
   - Actions shown: `status: { $ne: 'completed' }` (no frequency gate)
3. Momentum: 7-day vs prior 7-day message count → `rising` (>20% increase) / `falling` (<20% decrease) / `stable`.
4. Stage inference:
   ```
   openBlockers > 0 AND momentum.falling → 'blocked'
   lastDecision exists → 'discussion'
   momentum.recentMessageCount > 0 → 'ideation'
   else → 'ideation'
   ```
   NOTE: 'completed' and 'design' stages are in `Project.stage` enum but are NOT produced by `_inferStage`. They can only be set manually via `PATCH /api/projects/:projectId/stage`.
5. `pinnedContext`: compact text string (stage + momentum + top 3 decisions + top 3 blockers + top 6 topics + top 3 actions). Injected as Layer 1 of every AI system prompt.
6. `ProjectState.findOneAndUpdate({ projectId }, ..., { upsert: true })`.

## 10. FAILURE MODES

### Network failures

**MongoDB connection lost:** Mongoose reconnects automatically. During reconnection, HTTP requests → 500. WebSocket messages → `sendError`, message not saved. Extraction pipeline → caught, logged, ignored.

**Groq API unreachable:** `executeWithRetry` retries once for `ECONNRESET`/`ETIMEDOUT`/`ENOTFOUND`. After 2 failures: error shown in chat as System message.

**WebSocket connection lost (client):** `onclose` → `setTimeout(connectWebSocket, 3000)`. Frontend shows status banner. On reconnect: re-auth + re-join. NOTE: `ProjectWorkspace` creates raw WebSocket directly, NOT via `WebSocketService`. The service's message queue is unused.

### LLM failures

- Rate limit (Groq 429): not retried. Error shown in chat with `retryAfter` if available.
- Model not found (404): not retried.
- Invalid API key (401): not retried.
- Response contains no JSON (extraction): `parseExtractedJSON` returns empty. No entities extracted. No user-visible error.
- All entities fail validation: empty arrays stored. No user-visible error.

### Race conditions

**Concurrent extractions:** Both pass overlap guard (no artifacts yet), both call `mergeInsights`. MongoDB `findOneAndUpdate` with `upsert: true` is atomic — one creates, one updates. `occurrenceCount` may be incremented twice (acceptable over-counting).

**WebSocket reconnect during AI generation:** Client receives AI message in `discussion-joined` history AND as new `project-chat` broadcast → duplicate message visible. Frontend does not deduplicate.

### Partial writes

**Document upload with embedding failure:** Document saved, no chunks. Falls back to positional retrieval.

**Project creation with discussion failure:** Project exists without main thread. Appears in list but has no main thread.

### Silent failures

- Extraction pipeline failure: logged as `EXTRACTION_PIPELINE_FAILED (non-critical)`. User never sees it.
- Welcome email failure: logged. User registered successfully.
- `User.isOnline` staleness: set to `true` on login, never set to `false` on disconnect.
- `User.messageCount`: never incremented. Always 0.

## 11. NON-OBVIOUS BEHAVIORS

1. **Extraction only runs on main discussion.** `_triggerExtractionForMessage` fetches the discussion and returns early if `isMain === false`. Parallel discussions enter the knowledge model ONLY when a user explicitly creates a summary, which triggers the pipeline on the summary content.

2. **Parallel discussion summaries feed AI context.** `buildEntityAwareContext` fetches the latest summary for each parallel discussion and injects them as Layer 7 of the system prompt. A parallel discussion with no summary is invisible to the AI.

3. **Summarize is only accessible via three-dot menu.** The summarize button appears only in the three-dot dropdown of non-main discussions. Main thread three-dot menu shows "Discussion Summaries" (opens AllDiscussionSummaries view).

4. **Stale summary indicator.** Sidebar shows ⚠️ when `messageCount - latestSummary.messageCountAtSummary >= 10`. This means the discussion has grown significantly since the last summary and the AI may have outdated context.

5. **Dashboard refresh bypasses all gates.** `POST .../dashboard/refresh` calls `forceExtractForProject` which skips the rate gate and overlap guard. Only the sparsity guard applies (< 3 messages or < 150 chars → skip).

6. **`Topic` query bug in entity-aware context.** `Topic.find({ projectId, status: 'active' })` always returns empty because `'active'` is not a valid status enum value (valid: `candidate`, `stable`, `resolved`, `parked`). Layer 4 (topics) is always empty in the AI system prompt. The dashboard correctly uses `status: 'stable'`.

7. **`StrategicSignalEngine` is completely rewritten to read active data.** It no longer reads from `ProjectInsights`. It actively reads from `Decision`, `Blocker`, and `Topic` models to generate deep analytics (Decision Drift, Blocker Stagnation, Momentum Drop).

8. **Document embedding is async and invisible.** Upload response returns before embeddings exist. No UI indicator. Immediate `@CollabAI` query uses positional retrieval. The Documents view shows "⏳ Processing embeddings..." based on `doc.chunks.length === 0` — this is the chunk count from `documentService.getProjectDocuments` which queries `DocumentChunk.countDocuments`.

9. **Extraction counter resets on restart.** In-memory `extractionCounters` Map. After restart, first 4 messages are skipped, 5th triggers extraction.

10. **`WebSocketService` is dead code.** `frontend/src/services/websocket.js` defines a sophisticated service with reconnection and message queuing. `ProjectWorkspace.jsx` creates raw `new WebSocket(wsUrl)` directly. The service is never used.

11. **`AbortController` timeout does not cancel provider requests.** `callGroq`, `callOpenAI`, `callAnthropic`, `callGoogle` all ignore the signal parameter. The 90s timeout fires but the HTTP request continues.

12. **`TokenManager.trimContext` is never called.** Context overflow → error in chat, not graceful trimming.

13. **`Project.stage` and `ProjectState.stage` are independent.** `Project.stage` is manual via API. `ProjectState.stage` is computed automatically. They can diverge. `_inferStage` only produces `'blocked'`, `'discussion'`, or `'ideation'` — never `'design'` or `'completed'`.

14. **API keys stored in plaintext.** `Project.apiKeys` Map stores user-provided keys without encryption. The UI says "stored encrypted per project" — this is incorrect. `EncryptionService` is only used for the server's own Groq key.

15. **`EncryptionService` crashes server on startup.** If `ENCRYPTION_KEY` is not a valid 64-char hex string, the constructor throws and the server fails to start.

16. **Cross-project artifact contamination was a real bug (now fixed).** The extraction prompt previously contained specific technology examples. The LLM was returning these verbatim when conversations were sparse. Fixed by removing all specific examples from the prompt. The `clean-cross-project-artifacts.js` script can detect and remove any remaining contaminated artifacts.

17. **`discussionService.js` imports `User.js` directly.** This ensures the User schema is registered before any `populate('participants')` call. Required for standalone scripts that don't go through the full server module graph.

18. **`react-toastify` is used for all toasts.** The `ToastContext` and `useToast` hook exist but `App.jsx` uses `react-toastify`'s `toast` directly. `ToastContainer` is rendered in `App.jsx`.

## 13. SECURITY MODEL

### Secure
- Passwords: bcrypt 10 rounds. Never returned in responses.
- JWT: signed, 7-day expiry, verified on every request.
- CORS: restricted to `CORS_ORIGINS`.
- AI response rendering: `DOMPurify.sanitize(marked.parse(text))` — XSS protection.
- Project membership: all routes verify membership/ownership.

### NOT secure
- JWT secret fallback: two different hardcoded fallbacks in `authService.js` vs `config/index.js` if `JWT_SECRET` not set.
- Google OAuth direct endpoint: `POST /api/auth/google` without `idToken` accepts `googleId` + `email` without verification.
- API keys stored in plaintext in `Project.apiKeys` Map.
- Discussion access not enforced on backend: `GET /api/projects/:projectId/discussions` returns all discussions to any project member (frontend filters, backend does not).
- No CSRF protection (mitigated by Authorization header instead of cookies).
- No input length enforcement on WebSocket `text` field.
- Rate limiting bypassable: in-memory, resets on restart, keyed by `clientId` not `userId`.
- No audit log.

## 14. PERFORMANCE CHARACTERISTICS

### Bottlenecks

- **Groq API call:** 1–5 seconds. Dominant cost for every `@CollabAI` invocation.
- **KnowledgeAggregator dedup:** O(existing_entities × 384) embedding comparisons per new entity. For 50 existing decisions: ~50 embedding calls (~5–10 seconds) per extraction.
- **Dashboard message loading:** O(total messages across all discussions). No pagination. 10,000 messages → 10,000 documents loaded into memory per dashboard request.
- **VectorStore.search:** Loads ALL DocumentChunks for project into memory. O(n × 384) cosine similarity in JS loop. Linear in chunk count.
- **EmbeddingService first call:** Model initialization ~2–10 seconds. Subsequent calls ~50–200ms.

### Scalability limits

- Single server, no horizontal scaling. Rate limiters, extraction counters, WebSocket connections all in-memory.
- No message pagination in WebSocket flow (last 50 on join).
- Embedding model runs in-process (~100MB memory).
- Dashboard is O(total messages) — main scalability concern.
